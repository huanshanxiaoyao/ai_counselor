# MoodPal PRD（MVP 草案 v0.3）

## 1. 文档信息
- 文档状态：Draft（待产品/研发联合评审）
- 产品名称：MoodPal
- 中文名称：情绪管理伴侣
- 所属阶段：MVP（阶段一）
- 目标上线形态：Web
- 站内产品入口：统一命名为 `MoodPal`
- 站内子页面链接：`/moodpal/`

## 2. 背景与目标
### 2.1 背景
用户在日常情绪波动（焦虑、内耗、低落、关系受挫）时，缺少一个低门槛、随时可用、边界清晰的情绪支持入口。传统心理咨询成本较高、预约较重，不适合大量即时但非医疗级的问题场景。

### 2.2 产品目标（MVP）
1. 提供一个可以立即进入的情绪支持产品，让用户快速获得一次“被理解且有帮助”的对话体验。
2. 用虚拟角色承载不同流派的陪伴风格，让专业方法以轻松、亲和的形式被用户感知。
3. 通过多 Agent + 流派工作范式拆解，提升对话的专业性、稳定性与可解释性。
4. 在“隐私可控”前提下建立连续陪伴：仅保留用户确认后的摘要，原始对话会在会话结束后销毁。
5. 与现有站点能力对齐：支持匿名试用、token 消耗记录与配额控制、超额后联系管理员。

### 2.3 非目标（MVP 不做）
1. 不提供医疗诊断、药物建议或替代线下专业治疗。
2. 不做公开社区、用户间社交或内容广场。
3. 不做 token 公开售卖闭环，超额后仅支持联系管理员处理。
4. 不做多人群聊功能，群聊角色协作放入后续版本。

## 3. 目标用户与核心场景
### 3.1 用户类型
1. 匿名试用用户：希望“先用再说”，对门槛和隐私都较敏感。
2. 登录用户：希望形成连续陪伴，保留成长轨迹与会话摘要。

### 3.2 核心场景
1. 睡前焦虑、工作内耗、关系冲突后，需要快速获得情绪托底。
2. 用户希望被理解、被承接，而不是直接被教育或说教。
3. 用户希望在多次会话中围绕同一个问题持续推进，并在合适时机回顾行动尝试。

## 4. 产品原则
1. 极简进入：不做复杂前置测试，用户尽快进入对话。
2. 角色承载专业：前台看到的是亲和的角色，后台由具体流派方法支撑。
3. 轻松但专业：对话既要有人味，也要有方法，不是纯闲聊。
4. 隐私优先：原始消息不做长期保留，摘要需经用户确认后才归档。
5. 安全边界清晰：高风险场景优先切换至危机干预模式。
6. 不打扰：token 展示低打扰，但关键阈值要给出明确提醒与后续处理路径。

## 5. 功能需求（MVP）

### FR-01 角色选择（Persona Selection）
- 需求：用户进入 MoodPal 后，看到固定的 3 个虚拟角色卡片，选择一个角色进入对话。
- 设计原则：前台是角色人格与陪伴风格，后台映射到不同的流派方法与状态机工作范式。
- MVP 固定角色：
  1. 逻辑派的邻家哥哥：背后主要使用 CBT 工作方法，适合脑内反刍、担心搞砸、陷入负面假设的人。
  2. 共情派的知心学姐：背后主要使用人本主义/共情陪伴方法，适合情绪泛滥、需要被理解和承接的人。
  3. 深挖派的心理学前辈：背后主要使用更偏探索式的方法，适合反复卡在相似关系模式、希望理解深层触发因素的人。
- 卡片展示信息：
  1. 角色人设
  2. 角色风格
  3. 擅长解决的问题
  4. 简短介绍文案
- 验收标准：
  1. 首页至少展示 3 个固定角色卡片。
  2. 用户点击任一角色即可进入新会话。
  3. 文案中能让用户快速理解“这个角色适合帮助我处理什么问题”。

### FR-02 隐私契约（Privacy Handshake）
- 需求：每次新会话开始前，必须展示隐私契约与边界声明，用户确认后才可进入正式对话。
- 核心文案要点：
  1. 原始聊天记录不会长期保存。
  2. 会话结束后会自动生成摘要。
  3. 只有用户确认后的摘要才会被保留。
  4. 产品提供情绪支持，不构成医疗建议。
- 验收标准：
  1. 每次新会话都要求确认一次隐私契约。
  2. 用户未确认前，不允许进入正式会话。

### FR-03 核心对话流程（Core Conversation Flow）
- 需求：MoodPal 的核心体验是“由多个 Agent 协同、基于不同流派工作范式拆解”的专业对话流程。用户表面上是在和一个角色聊天，后台实际上由多 Agent 与状态机共同管理对话质量、流派方法执行、状态推进和安全边界。
- 产品层抽象流程：
  1. 用户输入当前困扰或情绪
  2. 系统识别用户当前意图、情绪状态与会话阶段
  3. 系统根据当前角色所对应的流派方法，选择合适的工作路径
  4. 系统调动相应的内部 Agent 完成分析、检索、提示词组装与回应生成
  5. 系统输出既符合角色风格、又符合流派方法的回复
  6. 系统更新当前会话状态，为后续继续推进做准备
- 具体说明：
  1. 如果用户选择的是 CBT 内核角色，则系统会沿着 CBT 的工作方法推进，例如识别情绪、识别自动化想法、梳理证据、尝试替代性认知、提出小行动。
  2. 如果用户选择的是共情型角色，则系统更强调情绪承接、反映、澄清与陪伴感。
  3. 技术层的多 Agent 与状态机细节不在本节展开，放入技术设计文档。
- 体验要求：
  1. 对话应体现角色一致性。
  2. 对话应体现流派方法，而不是完全自由发散。
  3. 用户应感受到“自然、轻松、被理解”，而不是在做生硬问卷。
- 验收标准：
  1. 系统能够根据所选角色保持稳定的对话风格。
  2. 对话能够体现对应流派的方法路径。
  3. 每轮回复后，会话状态会被推进或更新。

### FR-04 历史摘要与自然承接
- 需求：若用户存在上一次确认保存的摘要，新会话开始时可以基于摘要实现自然衔接。
- 规则：
  1. 登录用户按账号读取历史摘要。
  2. 匿名用户按 cookie 主体读取历史摘要。
  3. 若存在上次 homework，仅在用户情绪稳定时自然提及，不做审问式追问。
- 验收标准：
  1. 系统能读取最近一次已确认摘要并作为上下文。
  2. 历史动作项的提及方式保持自然、不施压。

### FR-05 危机识别与干预（Safety Guard）
- 需求：对自伤、他伤、自杀等高危表达进行实时检测，并强制切换安全干预模式。
- 地区策略：MVP 默认按中国大陆地区处理。
- 验收标准：
  1. 命中危机规则后，不再继续普通角色对话。
  2. 返回标准危机干预话术与中国大陆地区求助热线信息。
  3. 危机事件需有审计记录，但不保留用户敏感原文。

### FR-06 会话结束、摘要确认与阅后即焚
- 需求：会话结束后生成结构化摘要，用户可全文编辑，确认后归档；其余原始消息按“阅后即焚”原则销毁。
- 会话结束触发条件：
  1. 用户主动点击“结束会话”
  2. 用户连续 30 分钟无回复，系统自动判定会话结束
- 摘要字段（MVP）：
  1. 核心情绪状态
  2. 探讨焦点
  3. 重要触发因素/认知盲点
  4. 约定的微小行动（如有）
- 用户操作：
  1. 确认并归档
  2. 全文编辑后归档
  3. 全盘销毁
- 验收标准：
  1. 自动结束会话时，系统也会生成摘要。
  2. 用户可以对摘要进行全文编辑。
  3. 用户确认后，仅保留摘要；其他原始消息被销毁。
  4. 用户选择全盘销毁时，不保留摘要与原始消息。

### FR-07 轻量反馈
- 需求：摘要确认后，系统收集本次体验反馈。
- 反馈选项：
  1. 好多了
  2. 差不多
  3. 依然难受
- 验收标准：
  1. 反馈提交成功率 > 99%。
  2. 反馈结果可供后续角色匹配与产品分析使用。

### FR-08 匿名与登录策略
- 需求：默认允许匿名使用；匿名与登录用户都可形成连续陪伴。
- 规则：
  1. 匿名用户可直接进入 MoodPal。
  2. 匿名主体通过 cookie 标识。
  3. V1 允许匿名用户保留已确认摘要。
  4. 清理 cookie 后，V1 允许将其视为新主体。
  5. 登录用户的摘要归档与账号绑定。
- 验收标准：
  1. 匿名与登录用户都可启动会话。
  2. 匿名用户在不清 cookie 的情况下可读取自身历史摘要。

### FR-09 模型选择
- 需求：系统存在默认模型配置，同时允许用户在产品中选择使用模型。
- 验收标准：
  1. 用户进入对话时可看到默认模型或可选模型信息。
  2. 用户选择的模型会影响本次对话调用链路。
  3. 模型选择不改变角色与流派的产品语义。

### FR-10 token 消耗记录与配额控制
- 需求：记录每次调用 token 消耗，并向用户展示累积用量；超过配额后提示联系管理员。
- 配额策略：
  1. 测试期：100,000
  2. 上线后：1,000,000
- 运营承诺：
  1. 用户超额后可提交联系管理员反馈
  2. 运营 SLA：12 小时内响应
- 验收标准：
  1. 匿名和登录用户均记录 token 消耗。
  2. 全站有统一、低打扰的 token 使用展示。
  3. 超额时阻断新的 LLM 调用，并提供联系管理员入口。

### FR-11 合规与边界声明
- 需求：在产品首页和对话页都展示“非医疗建议”声明。
- 验收标准：
  1. 首页可见边界声明。
  2. 对话页也可见边界声明。
  3. 声明内容与危机干预逻辑保持一致。

## 6. 主流程（产品视角）
1. 用户进入 MoodPal 首页
2. 查看角色卡片并选择一个虚拟角色
3. 确认本次会话的隐私契约与边界说明
4. 进入正式对话
5. 后台基于角色对应流派，通过多 Agent + 状态机推进会话
6. 若触发危机规则，则立即切换至安全干预模式
7. 用户主动结束会话，或 30 分钟无回复自动结束
8. 系统生成摘要并展示给用户
9. 用户选择确认、全文编辑后保存，或全盘销毁
10. 系统销毁原始消息，仅保留允许保留的数据
11. 用户完成反馈，本次会话结束

## 7. 非功能需求
### 7.1 性能
1. 首屏可交互时间（P75）< 2.5s。
2. 对话请求端到端响应（P75）< 8s。
3. 上游模型超时或异常时，系统必须返回可理解的兜底反馈。

### 7.2 稳定性
1. 关键接口可用性目标：99.5%（MVP）。
2. 任意关键步骤失败时，不出现裸 500 或空白页。

### 7.3 安全与隐私
1. 不在日志中记录用户原始敏感文本。
2. 会话结束后的销毁流程可追踪，但审计内容只记录事件，不记录原文。
3. 全量接口遵守现有鉴权、CSRF、防滥用策略。

## 8. 核心数据对象（产品视角）
1. `Session`：一次完整会话，含角色、状态、开始/结束时间。
2. `SessionState`：当前会话所处阶段与内部工作状态。
3. `SessionSummary`：用户确认后的结构化摘要。
4. `CrisisEvent`：危机触发事件（脱敏元数据）。
5. `TokenUsage`：token 消耗流水。
6. `QuotaFeedback`：超额后联系管理员的反馈记录。

## 9. 埋点与效果评估（MVP）
### 9.1 核心漏斗
1. 进入产品页 -> 选择角色 -> 确认隐私契约 -> 完成首轮对话 -> 完成摘要确认

### 9.2 关键指标
1. 首次会话完成率
2. 摘要确认率
3. 7 日回访率
4. homework 被提及率
5. 危机触发后引导页到达率
6. 超额用户反馈提交率

## 10. 里程碑建议
1. M1（1 周）：产品信息架构、角色定义、首页与会话页骨架。
2. M2（1-2 周）：核心对话主流程、多 Agent/状态机基础链路、危机拦截、摘要确认闭环。
3. M3（1 周）：token 配额联动、埋点监控、灰度发布与体验打磨。

## 11. 验收清单（发布门槛）
1. 3 个固定角色均可独立发起会话。
2. 每次新会话都必须完成隐私契约确认。
3. 核心对话能体现角色风格与流派方法。
4. 30 分钟无回复时可自动结束并生成摘要。
5. 摘要支持全文编辑后保存。
6. 阅后即焚流程能正常执行，仅保留允许保留的数据。
7. 危机命中后可稳定切换至中国大陆地区安全干预话术。
8. 匿名与登录用户都能记录 token、读取允许保留的摘要、触发超额联系管理员流程。
9. 首页和对话页都能看到非医疗建议声明。

## 12. 仍需继续讨论的问题
1. 3 个角色的视觉设定、头像风格和对外文案是否需要单独产出角色设定卡。
2. 默认模型与可选模型的前台展示方式，是否要区分“推荐模型”和“高级选项”。
3. 自动结束会话后进入摘要页的交互细节，是否需要额外提示“本次会话已结束且不可恢复”。
