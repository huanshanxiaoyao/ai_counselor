# 圆桌会谈 (Roundtable Discussion)

## 1. 概念与愿景

**一句话描述：** 一个 AI 驱动的多角色圆桌讨论平台，让历史人物围坐一桌，就用户指定的话题展开深度对话。

**核心体验：** 用户扮演主持人或参与者，与多个 AI 角色进行结构化、有节制的讨论。每个角色都有独特的背景、观点和语言风格，由"导演"AI 根据话题生成角色设定。

**差异化价值：**
- 角色设定的深度还原：导演 Agent 根据话题为每个角色生成详尽的背景设定、语言风格、知识边界
- 结构化的讨论节奏：主持人 AI 把控全局，避免冷场和跑题
- 真实的圆桌体验：发言权机制模拟真实圆桌会议的礼貌与秩序
- 严谨的时代意识：确保角色只讨论其所在时代已知的事物

## 2. 产品结构

### 2.1 页面结构

```
/
├── /                           # 首页（工具索引）
└── /roundtable/               # 圆桌会谈
    ├── /                       # Step 1: 话题设置
    ├── /setup/                 # Step 2: 角色配置
    ├── /discussion/            # Step 3: 讨论进行中
    └── /summary/               # Step 4: 讨论总结
```

### 2.2 路由设计

| 路由 | 功能 |
|------|------|
| `/roundtable/` | 话题输入 & 基本配置 (Step 1) |
| `/roundtable/setup/` | 角色配置 (Step 2) |
| `/roundtable/d/<id>/` | 讨论进行 (Step 3) |
| `/roundtable/characters/` | 角色设定管理页面 |

### 2.3 API 设计

| API | 功能 |
|-----|------|
| `POST /roundtable/api/suggestions/` | 获取导演 Agent 推荐的 20 个候选角色 |
| `POST /roundtable/api/configure/` | 为选定角色生成详细配置 |
| `POST /roundtable/api/start/` | 创建讨论并生成开场 |
| `WS /ws/roundtable/d/<id>/` | WebSocket 进行实时讨论 |
| `GET /roundtable/api/history/` | 获取历史讨论列表 |
| `POST /roundtable/api/restart/<id>/` | 复制配置重新开始讨论 |

## 3. 功能详述

### 3.1 话题设置 (Setup) - Step 1

**输入字段：**
- **话题** (必填): 简短描述讨论主题，如"项羽该不该渡江"
- **用户角色**: 主持人 / 参与者 / 旁观者

**注意：**
- 选择「主持人」时，系统会提醒用户需要手动 @ 角色发言，角色不会主动发言

---

#### 导演 Agent (Director Agent)

**核心职责：** 根据话题，通过 LLM 生成候选角色列表

**触发时机：** 用户输入话题并点击"获取推荐角色"时

**工作流程：**

```
┌────────────────────────────────────────────────────────────┐
│ Step 1: 话题分析                                            │
│ - 解析话题意图（哲学思辨？历史评析？文学讨论？管理决策？）      │
│ - 识别话题涉及的核心领域、时期、思潮                          │
│ - 判断需要什么背景、性格、观点的角色参与                      │
└────────────────────────────────────────────────────────────┘
                              ↓
┌────────────────────────────────────────────────────────────┐
│ Step 2: 角色推荐                                            │
│ - 调用 LLM，根据话题推荐 20 个真实历史人物                   │
│ - 必须确保多样性：不同背景、不同立场、不同时代                │
│ - 返回格式：[{name, era, brief_reason}, ...]                │
└────────────────────────────────────────────────────────────┘
```

**导演 Agent 推荐角色的 Prompt 示例：**

```
你是一位经验丰富的导演，负责为圆桌讨论挑选最合适的角色。

请根据以下话题，推荐 20 位真实的历史人物参与讨论：

话题：{user_topic}

要求：
1. 必须全部是真实存在的历史人物，不能是虚构角色或纯文学人物
2. 确保多样性：
   - 不同背景：哲学家、政治家、文学家、科学家、军事家等
   - 不同立场：支持方、反对方、中立方
   - 不同时代：古代、近代、现代（允许跨时空讨论）
3. 每位人物需说明推荐理由（为什么这个人物适合这个话题）

请以 JSON 数组格式返回，示例：
[
  {"name": "孔子", "era": "春秋", "reason": "儒家思想代表，对仁义有深刻见解"},
  {"name": "韩非子", "era": "战国", "reason": "法家思想代表，与儒家形成对比"}
]

只返回 JSON，不要有其他解释。
```

**多样性校验规则：**

1. **背景多样性**：推荐的角色应涵盖不同领域

   - 政治家/统治者
   - 军事家
   - 文学家/学者
   - 改革家/革命者

2. **立场多样性**：同一话题应有不同观点
   - 支持方代表
   - 反对方代表
   - 中立/调和派

3. **时代跨度**：
   - 允许跨时空讨论，不同历史时期的人物可以同台
   - 优先选择与话题时代相关的角色

### 3.2 角色配置 - Step 2

**前置条件：** 用户已确定最终参与的 5 个角色（可增删调整）

**核心目标：** 为每个选定角色生成详细的讨论设定

---

#### 3.2.1 角色版本判定

**判定规则：** 灵活把握，对历史上早年和晚年差异较大的人物进行细分，否则不区分

**需要细分版本的人物示例：**

| 人物 | 建议版本 |
|------|---------|
| 诸葛亮 | 卧龙初期（27岁前）/ 蜀汉丞相时期 / 五丈原临终前 |
| 曹操 | 青年仕宦期 / 魏王期 |
| 苏轼 | 青年意气 / 晚年豁达 |
| 曾国藩 | 青年翰林 / 湘军统帅 / 晚年洋务 |

**不需细分的人物示例：**
- 苏格拉底（一生相对单一）
- 马克思（一生围绕理论研究）
- 鲁迅（思想脉络清晰）

**版本判定 Prompt：**

```
判断以下历史人物是否需要区分不同人生阶段版本：

人物：{name}
时代：{era}

请判断：
1. 该人物的人生早期和晚期是否有显著差异？
2. 如果有，建议分为几个版本，每个版本的简要描述

请以 JSON 格式返回：
{
  "need_versioning": true/false,
  "versions": [
    {"version": "版本名", "description": "版本描述"}
  ]
}
```

#### 3.2.3 角色配置流程

```
┌────────────────────────────────────────────────────────────┐
│ 触发时机                                                    │
│ 用户点击"开始讨论"时，对最终选定的 5 个角色执行              │
└────────────────────────────────────────────────────────────┘
                              ↓
┌────────────────────────────────────────────────────────────┐
│ Step 1: 版本判定                                            │
│ - 判断是否需要细分早/晚年版本                                │
│ - 如需细分，生成多个版本配置                                 │
│ - 如不需细分，只生成一个通用版本                             │
└────────────────────────────────────────────────────────────┘
                              ↓
┌────────────────────────────────────────────────────────────┐
│ Step 2: 基础信息生成（每个角色版本调用 1 次 LLM）           │
│ - 扩展 bio（100字以内）                                    │
│ - 生成 background（200字以内）                             │
│ - 生成 major_works（2-3 项）                              │
│ - 生成 viewpoints（3-5 个维度）                            │
│ - 生成 temporal_constraints                                │
└────────────────────────────────────────────────────────────┘
                              ↓
┌────────────────────────────────────────────────────────────┐
│ Step 3: 语言风格生成（每个角色版本调用 1 次 LLM）           │
│ - 生成 language_style                                       │
│   - tone: 语气特点                                        │
│   - catchphrases: 常用表达 3-5 个                         │
│   - speaking_habits: 说话习惯                            │
└────────────────────────────────────────────────────────────┘
                              ↓
┌────────────────────────────────────────────────────────────┐
│ Step 4: 代表文章关联（每个角色版本调用 1 次 LLM）           │
│ - 根据 topic，推荐 1-2 篇与话题相关的代表文章/著作           │
│ - 优先从经典文本中选择                                       │
│ - 若原文超过 200 字，由 LLM 摘要压缩                       │
│ - 格式：[{"title": "...", "source": "...", "content": "...", "reason": "..."}] │
└────────────────────────────────────────────────────────────┘
                              ↓
┌────────────────────────────────────────────────────────────┐
│ Step 5: 整合输出                                            │
│ - 汇总所有生成的内容                                         │
│ - 检查是否有矛盾或缺失                                       │
│ - 输出最终角色配置                                           │
└────────────────────────────────────────────────────────────┘
```

**总计：每个角色版本需要 3 次 LLM 调用，5 个角色约 15 次调用**

#### 3.2.4 角色配置 Prompt 示例

**Step 2: 基础信息（含时代边界）**

```
请为以下历史人物生成适合圆桌讨论的背景信息：

人物：{name}
时代：{era}
讨论话题：{topic}

请生成：
1. bio：一句话简介（50字以内）
2. background：详细背景介绍（200字以内），包括：
   - 人物生平概要
   - 与讨论话题相关的经历
   - 在该话题上的立场基础
3. major_works：2-3 项主要作品/成就
4. viewpoints：针对话题的 3-5 个核心观点维度
5. temporal_constraints：时代认知边界
   - can_discuss：该人物可能讨论的话题/概念（其所在时代已知的事物）
   - cannot_discuss：该人物不可能知道的事物（如：后世朝代名称、近现代概念）
   - knowledge_cutoff：该人物知识的最终时间点

请用 JSON 格式返回。
```

**Step 3: 语言风格**

```
请为以下历史人物生成语言风格描述：

人物：{name}
时代：{era}
人物背景：{background}

请生成该人物在圆桌讨论中的语言风格：
1. tone：整体语气特点（如：豪迈直率、温文尔雅、犀利冷峻）
2. catchphrases：3-5 个该人物可能会用的经典表达
3. speaking_habits：说话习惯（如：善用反问、惯用排比、言简意赅）

请用 JSON 格式返回。
```

**Step 4: 代表文章**

```
讨论话题：{topic}
人物：{name}
时代：{era}

请为该人物推荐 1-2 篇与讨论话题相关的代表文章/著作片段：

规则：
1. 优先选择该人物本人的著作
2. 如无本人著作，选择描述该人物的重要文献
3. 如原文超过 200 字，请进行摘要压缩，保留核心观点
4. 需要标注文章来源（source）

请用 JSON 格式返回：
[
  {
    "title": "文章/著作标题",
    "source": "《作品名》作者（可选）",
    "content": "原文或摘要（200字以内）",
    "reason": "为什么推荐这篇"
  }
]
```

#### 3.2.5 时代边界约束（重要）

**核心原则：** 角色只能讨论其所在时代已知的事物

**约束规则：**

| 角色时代 | 不能讨论的内容示例 |
|---------|-----------------|
| 先秦 | 儒家思想成熟体系、科举制度 |
| 汉代 | 魏晋南北朝以后的历史事件 |
| 三国 | 晋朝建立、隋唐盛世 |
| 宋代 | 元明清的历史、清朝闭关锁国 |
| 清代 | 辛亥革命、五四运动 |
| 民国 | 新中国成立、改革开放 |
| 现代（1949前）| 互联网、AI、智能手机 |
| 现代（1949后）| 可以讨论当代话题 |

**Prompt 中需要明确告知 LLM：**

```
【重要时代约束】
该角色生活在 {era}，其知识截止于 {knowledge_cutoff}。
在发言时，该角色：
- 只能提及其生前已知的事物、历史事件、人物
- 不得提及尚未发生的历史（如不能让清朝人谈论民国）
- 不得使用近现代才出现的概念和词汇（如不能让古人谈"AI"）
- 如果被问及相关话题，应表示"不知"或"未曾听闻"
```

---

#### 用户操作

在角色配置过程中，用户可以：

- **增删角色**：从 20 个候选角色中选择 5 个参与讨论
- **调整出场顺序**：拖拽调整发言顺序
- **选择角色版本**：如果角色有多个版本，用户可选择具体版本
- **跳过/精简配置**：如果配置时间过长，可选择简化模式

**LLM Provider 选择：**
- 是独立于角色的全局配置
- 在"讨论设置"中统一选择
- 可选：每个角色的 LLM Provider（进阶功能，默认使用统一配置）

│ │ [                                          ] [发送]   │ │
│ │                                                        │ │
│ │ [📋 待发言角色: 秦始皇] [👀 旁观者模式] [🔄 12/100轮]  │ │
│ └───────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘

#### 发言权机制

**核心规则：**

1. **主持人控制发言权**
   - 只有主持人 @ 某个角色，该角色才能发言
   - 用户作为主持人时，自己控制发言节奏
   - AI 主持人时，用户可请求主持人 @ 某个角色

2. **发言格式**
   ```
   [角色名]: @主持人 [发言内容]

   或（需要先获得发言权）

   [角色名]: [发言内容] @主持人
   ```

3. **字数限制**
   - 所有参与者（包括用户）**统一限制 200 字**
   - 超出限制的内容将被截断

4. **发言结束标记**
   - 角色发言以 `@主持人` 结束
   - 表示发言权交回主持人

---

#### 用户角色行为

| 用户角色 | 行为描述 |
|---------|---------|
| **主持人** | 全程控制发言权，手动 @ 各角色邀请发言，可随时打断或总结 |
| **参与者** | 与 AI 角色同等地位，需要向主持人请求发言权，受 200 字限制 |
| **旁观者** | 只能观看，不能发言，可随时切换为参与者模式 |

**角色切换规则：**
- 用户可在讨论过程中自由切换角色身份
- 如果用户从「主持人」切换到其他角色，**AI 主持人 Agent 自动接管**主持工作
- AI 主持人接管后，继续按照主持人职责运行（控场、授权、引导）
- 如果用户切换回「主持人」，可随时从 AI 主持人手中收回主持权

---

#### 讨论进度

- **最大轮次限制：** 20 轮
- **进度显示：** 显示当前轮次，如 `12/20`
- **20 轮后：** 自动触发讨论结束流程

---

### 3.4 主持人 Agent (Moderator)

**重要说明：**
- **每场讨论都必须生成 AI 主持人 Agent**
- 无论用户选择何种角色，AI 主持人 Agent 在后台待命
- 只有当用户主动选择「主持人」角色时，AI 主持人才会在后台待命
- 当用户退出「主持人」角色时，AI 主持人立即接管

**职责列表：**

| 职责 | 触发条件 | 行为 |
|------|---------|------|
| 开场 | 讨论开始（仅用户非主持人时） | 介绍话题、参与者、规则，邀请第一位角色 |
| 控场 | 持续 | 确保讨论不跑题，平衡各方发言机会 |
| 授权 | 用户/角色请求 | 决定是否授予发言权 |
| 引导 | 冷场/跑题 | 适时提问或转移话题 |
| 阶段性总结 | 每 5-10 轮后 | 总结已讨论要点 |
| 收尾 | 用户结束或达到 100 轮 | 生成讨论总结 |
| 自动接管 | 用户退出主持人角色 | 立即接管主持工作 |

**主持人说话风格：**
- 中立：绝不表达自己的观点
- 引导性：多用提问推动讨论
- 简洁：发言简短有力
- 礼貌：维护良好讨论氛围

---

### 3.5 讨论总结 - Step 4

**生成内容：**

| 内容 | 说明 |
|------|------|
| 话题概述 | 一段话总结讨论主题 |
| 各方主要观点 | 每个角色的核心论点 |
| 关键分歧点 | 角色之间观点冲突的地方 |
| 金句摘录 | 讨论中的精彩发言（3-5 条） |
| 讨论贡献统计 | 每个人的发言次数、字数统计 |

## 8. 风险与挑战

### 8.1 角色一致性
**问题：** 如何确保 AI 角色始终保持设定一致？

**解决方案：**
- 详细的 system prompt 包含完整角色设定
- 字数限制减少偏离概率
- 主持人 Agent 适时纠正跑偏发言

### 8.2 讨论深度
**问题：** 避免浅尝辄止的泛泛而谈

**解决方案：**
- 主持人 Agent 主动追问
- 阶段性总结引导深入
- 代表文章提供专业素材

### 8.3 时代边界一致性
**问题：** 如何确保角色只讨论其时代已知的事物？

**解决方案：**
- 在 system prompt 中明确标注 temporal_constraints
- 代表文章帮助角色保持时代感
- 如果角色提及未来内容，主持人及时纠正
- LLM 调用时明确告知角色生活的时代和知识截止点

### 8.4 LLM 延迟
**问题：** 多角色配置调用可能很慢

**解决方案：**
- 显示配置进度（"正在配置角色 3/5..."）
- 可选的简化配置模式
- 角色配置可缓存复用

### 8.5 话题敏感性
**问题：** 讨论可能触及敏感历史话题

**解决方案：**
- 用户需对话题自行负责
- 角色发言过滤（基础敏感词过滤）
- 主持人 Agent 可跳过不当发言
