# MoodPal 全能主理人 PRD（草案 v0.3）

## 1. 文档定位
- 文档对象：MoodPal 新增角色“全能主理人”
- 文档目标：定义这个角色的产品定位、用户价值、主流程、流派切换原则与验收边界
- 与主 PRD 的关系：本文件是 [prd.md](/Users/suchong/workspace/ai_counselor/docs/moodpal/prd.md) 的专题子文档；未特别说明的隐私、摘要、危机、匿名主体、token 配额、模型选择、非医疗声明等，均继承主 PRD
- 文档状态：Draft，待产品确认后再进入技术设计

## 2. 为什么要做“全能主理人”
### 2.1 问题
当前 MoodPal 已有 3 个明确角色：
1. 逻辑派的邻家哥哥（CBT）
2. 共情派的知心学姐（人本主义）
3. 深挖派的心理学前辈（精神分析式探索）

这 3 个角色已经覆盖了不同方法路径，但用户在第一次进入产品时，往往并不知道自己更适合哪一种方式。直接让用户做方法选择，会带来三类问题：
1. 选择成本高：不知道该选谁
2. 错配风险高：明明只是需要先被接住，却选到了偏问题拆解的角色
3. 体验割裂：用户以为要重新开新会话，才可以换一种支持方式

### 2.2 产品目标
“全能主理人”要解决的不是“替代三个角色”，而是：
1. 作为默认推荐入口，降低初次使用的选择焦虑
2. 先稳住用户，再根据状态动态选择更合适的工作方式
3. 让用户感受到始终在和同一个角色说话，但后台可以调用不同流派能力
4. 在不增加用户理解负担的前提下，提高首会话完成率和角色匹配准确度

### 2.3 非目标
MVP 不把“全能主理人”做成以下形态：
1. 不做“专家会诊式多角色群聊”
2. 不做对用户显式展示的复杂流派切换面板
3. 不做医疗级分诊或诊断建议
4. 不承诺“自动选出唯一正确流派”，而是追求更稳妥、更合适的起步与推进

## 3. 角色定位
### 3.1 用户看到的角色
- 角色名：全能主理人
- 内部代号：Master Guide
- 推荐定位：默认推荐
- 角色气质：沉稳、灵活、可靠，不急着分析，也不会只停留在表面安慰
- 角色承诺：先接住你，再帮你判断此刻更适合被理解、被梳理，还是去看更深的重复模式
- 轻量说明口径：允许在入口或会话开场用一句自然文案提示“我会根据你的状态调整陪伴方式”，但不显式展示后台流派切换面板

### 3.2 与其他 3 个角色的关系
1. 全能主理人是新的产品入口，不替代现有 3 个固定角色
2. 现有 3 个角色仍然保留，适合有明确偏好的用户直接选择
3. 全能主理人本质上是“统一前台角色 + 后台动态路由”的整合模式
4. 用户在全能主理人会话中，不需要理解后台具体切到了哪条流派链路

### 3.3 首页入口定义
MVP 确认按以下方案定义：
1. 在 `/moodpal/` 首页增加第 4 张角色卡：全能主理人
2. 该卡片置于首位，并加“默认推荐”标识
3. 卡片文案重点不是强调流派，而是强调“如果你不确定该选谁，就先来这里”

卡片文案草案：
- 标题：全能主理人
- 副标题：默认推荐｜整合式陪伴
- 描述：如果你不确定自己现在更需要被接住、被梳理，还是想看清反复出现的模式，可以先从这里开始。
- 擅长问题：刚进来不知道选谁、情绪和问题交织在一起、希望有人边听边帮你判断方向

## 4. 核心产品原则
1. Holding First：每次新会话都先以人本主义式抱持开局，不允许一上来就冷启动为 CBT 拆解或精神分析解释
2. Humanistic as Base Layer：人本主义不是和 CBT / 精神分析平权竞争的主轨，而是承担开场承接、过程修复、关系缓冲的底层支撑
3. Two Main Tracks：真正的主工作轨是 CBT 与精神分析；其中精神分析触发门槛更高、通常更偏后置
4. One Persona Surface：前台始终是全能主理人一个角色，不暴露技术层图谱和节点名；对用户只做轻量说明，不做显式流派切换展示
5. Per-turn Single Main Track：针对用户的一次输入，系统只确定一个“当轮主工作方式”；人本主义可作为开场承接或修复层存在，但同一条回复中不并列执行 CBT 和精神分析两种主干干预
6. Necessary Switching：单次会话允许多次主轨切换，但每次切换都必须有明确理由，不能为了“看起来聪明”而乱跳
7. Repair before Push：一旦出现联盟裂痕、阻抗升高或情绪失稳，优先修复关系，而不是继续推进深层工作
8. Same Privacy Contract：无论后台走哪条流派链路，摘要确认与阅后即焚规则完全一致

## 5. 目标用户与适用场景
### 5.1 核心用户
1. 第一次使用 MoodPal，不知道该选哪个角色的人
2. 当前困扰既有情绪，也有现实问题，不知道应该先安慰还是先梳理的人
3. 觉得自己好像总在重复某种模式，但又不确定是否适合直接深挖的人

### 5.2 典型场景
1. 用户刚经历冲突，情绪很满，但同时又想知道“接下来怎么办”
2. 用户表面说的是某个现实问题，聊着聊着出现明显的重复性关系模式
3. 用户起初只想倾诉，在被接住后，逐渐具备拆解问题或探索模式的准备度

### 5.3 不适合场景
1. 用户已经明确知道自己只想使用某个固定角色
2. 用户希望获得临床诊断、药物建议或强干预结论
3. 高危危机场景，必须直接进入安全干预，不继续常规整合式对话

## 6. 功能需求

### FR-MG-01 角色入口与选择
- 需求：MoodPal 首页提供“全能主理人”角色卡，作为默认推荐入口
- 规则：
  1. 用户可像其他角色一样直接点击进入新会话
  2. 首页文案明确说明它是“帮你判断更适合哪种支持方式”的入口
  3. 允许用一句轻量文案提示“会根据你的状态调整陪伴方式”
  4. 不取消原有 3 个角色卡
- 验收标准：
  1. 首页存在第 4 张角色卡
  2. 该卡片带默认推荐标记
  3. 点击后进入新的会话启动流程

### FR-MG-02 会话前隐私契约
- 需求：全能主理人会话与其他角色一致，每次新会话开始前必须确认隐私契约
- 规则：
  1. 沿用主 PRD 已有隐私契约与非医疗声明
  2. 不因其具备动态分诊能力而新增额外授权步骤
- 验收标准：
  1. 用户未确认前，不能进入正式对话
  2. 确认后才创建会话

### FR-MG-03 开场与修复层：人本主义托底
- 需求：全能主理人的人本主义层负责新会话开场承接，以及主轨推进过程中的关系修复与情绪缓冲
- 规则：
  1. 首轮回复必须体现承接、合法化、情绪托底
  2. 首轮不直接进入冷启动议程设定、证据辩论或深层解释
  3. 当用户在对话过程中出现失稳、被冒犯感、联盟裂痕或明显抽离时，应优先回到人本主义修复层
  4. 人本主义层既可以单独成为某一轮的全部工作方式，也可以作为主轨回复前的短承接，但不负责同时输出两种主干方法
  5. 若用户一上来就是高危表达，则本规则被危机干预抢占
- 验收标准：
  1. 非危机场景下，首轮回复语气明显偏抱持而非分析
  2. 首轮不会直接出现过强的解决导向或解释导向
  3. 用户失稳时，系统能够退回人本主义修复层，而不是硬推原主轨

### FR-MG-04 动态分诊与主工作轨路由
- 需求：在建立基本安全感后，系统根据用户状态在“人本主义层 / CBT 主轨 / 精神分析主轨”之间动态调度，但其中只有 CBT 与精神分析是主工作轨
- 路由结构：
  1. 安全抢占层：危机内容优先级最高，直接中断常规对话
  2. 人本主义层：负责开场承接、关系修复、情绪缓冲、主轨切换前后的柔性过渡
  3. 主工作轨：`CBT` 与 `精神分析`，二者承担主要的问题推进与探索工作
- 当轮路由规则：
  1. 针对用户的一次输入，只确定一个“当轮主工作方式”
  2. 如果当前更需要被接住或修复关系，则当轮只做人本主义层
  3. 如果当前适合推进问题工作，则当轮主工作方式在 `CBT` 与 `精神分析` 中二选一
  4. 同一条回复中，不并列执行 CBT 和精神分析两种主干干预；下一轮可基于用户反应重新评估
- CBT 触发原则：
  1. 用户开始呈现具体事件、明确问题、现实困境或决策压力
  2. 用户表现出一起拆解、梳理、尝试行动的意愿
  3. 当前问题更像“怎么想、怎么做、怎么应对”的卡点，而不是必须先看长期重复模式
- 精神分析触发原则：
  1. 用户持续出现“总是、每次、反复”之类的长线模式信号
  2. 对话中已经积累了足够材料，联盟相对稳定，用户具备一定探索耐受度
  3. 精神分析通常比 CBT 更偏后触发，不应在证据不足、联盟未建立或用户仍明显失稳时贸然进入
- 多次切换原则：
  1. 单次会话允许多次主轨切换，不要求“一条主轨走到底”
  2. 每次切换都必须有明确触发原因，例如情绪稳定度变化、问题清晰度变化、重复模式显现、用户从探索转向行动、或出现阻抗/联盟裂痕
  3. 不允许因为一句话的轻微措辞波动就切轨；切换必须基于连续信号或高优先级事件
  4. 切换后应至少完成一个有意义的推进动作，再决定是否继续切换；若推进失败，优先退回人本主义层
- 典型允许路径：
  1. 人本主义开场 -> CBT -> 人本主义修复 -> CBT
  2. 人本主义开场 -> 精神分析 -> 人本主义修复 -> 精神分析
  3. 人本主义开场 -> CBT -> 精神分析：当具体问题被梳理后，进一步显露长期重复模式
  4. 人本主义开场 -> 精神分析 -> CBT：当已经看到模式，用户开始希望落到现实行动与应对
- 用户体验要求：
  1. 路由切换不应改变“角色像换了一个人”的感觉
  2. 切换后仍保持全能主理人的统一口吻
- 验收标准：
  1. 系统能基于用户状态判断当轮应停留在人本主义层，还是进入 CBT / 精神分析主轨
  2. 切换过程对用户是平滑的
  3. 对话中允许多次切换，但不会出现无理由的抖动式切换
  4. 同一条回复中不会并列执行 CBT 与精神分析两种主干干预
  5. 在一次会话内，系统能够根据状态变化完成至少一类跨主轨切换场景

### FR-MG-05 阻抗降级与关系修复
- 需求：当后台已经进入 CBT 或精神分析主轨后，如检测到用户失稳、抵触或联盟裂痕，应及时降级回人本主义修复层
- 触发信号示例：
  1. 用户表达“你根本不懂我”“别分析我了”
  2. 用户明显愤怒、防御、否定、沉默或抽离
  3. 用户被追问后情绪显著恶化
- 处理原则：
  1. 先修复关系，再决定是否继续推进
  2. 修复后可继续留在人本主义层，也可在后续重新评估是否回到 CBT / 精神分析主轨，或切到另一条更合适的主轨
  3. 不在裂痕未修复时强行推进原主轨任务
- 验收标准：
  1. 命中阻抗或联盟裂痕后，系统能及时停止深推
  2. 回复转为承接和修复语气

### FR-MG-06 危机干预优先级
- 需求：全能主理人不改变现有安全规则；危机检测永远先于普通流派路由
- 规则：
  1. 每轮用户输入先过危机检测
  2. 一旦命中高危规则，直接退出常规整合式对话
  3. 用户看到的是统一的危机干预响应，而不是继续角色化深入对话
- 验收标准：
  1. 危机抢占优先于人本/CBT/精神分析路由
  2. 危机事件按主 PRD 的脱敏原则记录

### FR-MG-07 历史摘要与连续陪伴
- 需求：全能主理人也要复用 MoodPal 的摘要记忆能力，并能自然承接用户上次会话
- 规则：
  1. 读取主体最近一次已确认摘要，来源可以是任意角色会话
  2. 历史承接应以自然关心方式带出，不做“系统复盘式盘问”
  3. 若历史摘要已显示某用户更适合某类工作方式，本次可以更快进入对应主轨，但仍先做人本主义开场承接
- 验收标准：
  1. 新会话可读取最近一次已确认摘要
  2. 承接文案自然，不破坏角色感

### FR-MG-08 摘要确认与阅后即焚
- 需求：全能主理人沿用现有摘要确认与 Burn Pipeline，但摘要要能反映“整合式推进”的结果
- 用户可见摘要建议至少包含：
  1. 本次主要情绪状态
  2. 本次讨论焦点
  3. 本次已尝试的支持方式与最终更适合的工作方向
  4. 看见的关键触发点 / 重复模式 / 微小行动（如有）
- 规则：
  1. 用户可全文编辑后保存
  2. 摘要对用户保持自然语言表达，不暴露内部技术节点名，但可以自然表达“本次先被接住，再一起梳理问题，后来也看到某个重复模式”
  3. 原始消息仍按主 PRD 在确认后销毁
  4. 若用户选择全盘销毁，则 session 内的 `route_trace`、`summary_hints` 与相关编排状态也应一并清空，不保留可回放的单会话切轨轨迹
  5. 若业务需要产品分析，只允许保留不可逆的聚合统计，例如切换次数、是否触发过某主轨；这些统计不应携带用户原文，也不应用于重建单个会话轨迹
- 验收标准：
  1. 用户确认后，仅保留摘要
  2. 摘要能体现本次“接住 + 分诊 + 推进”的结果
  3. 用户选择全盘销毁后，系统不保留 session 级可回放路由轨迹

### FR-MG-09 模型与 token 配额继承
- 需求：全能主理人继承 MoodPal 现有模型选择、token 消耗记录、配额提醒与管理员反馈能力
- 特别要求：
  1. 后台即使发生流派切换，也仍记录到同一主体和同一会话维度
  2. token 展示保持低打扰，不因新增角色而加重 UI 复杂度
- 验收标准：
  1. 使用全能主理人时也能正常记录 token
  2. 超额时仍按现有站点逻辑提示联系管理员

### FR-MG-10 与专属角色的协同
- 需求：全能主理人是默认推荐入口，但不阻断用户后续转向专属角色
- MVP 约束：
  1. 本次会话内，不做“显式一键切换成另一个前台角色”
  2. 用户如有明确偏好，仍可回到首页直接新开对应专属角色会话
  3. 后续版本可评估是否增加“推荐你下次直接找某个角色”的温和引导
- 验收标准：
  1. 全能主理人与专属角色在产品信息架构上并存
  2. 不会造成用户误以为原有 3 个角色已被替代

## 7. 主流程（产品视角）
1. 用户进入 `/moodpal/`
2. 在首页看到“全能主理人”默认推荐卡，以及原有 3 个专属角色卡
3. 用户点击全能主理人，确认隐私契约
4. 系统创建新会话，读取最近一次已确认摘要作为可选上下文
5. 首轮回复以人本主义式承接开场
6. 后台持续评估：
   - 是否需要先停留在人本主义层做承接或修复
   - 若适合推进，本轮主工作轨应选择 CBT 还是精神分析
   - 是否在后续轮次基于新信号切换主工作轨
7. 如出现危机、阻抗或联盟裂痕，优先抢占或退回人本主义修复层
8. 用户主动结束，或超时自动结束
9. 系统生成整合式摘要，必要时体现本次支持方式的阶段性变化
10. 用户确认保存、全文编辑后保存，或全盘销毁
11. 原始消息销毁，仅保留允许保留的数据

## 8. 体验要求
1. 用户不需要理解后台流派切换，也不应被技术术语打扰
2. 开场要温和，切换要自然，收尾要清楚
3. 同一会话里，全能主理人的人设必须稳定
4. 即使后台切了流派，口吻也要统一，不出现明显“人格突变”
5. 当系统判断失配时，要优先保守，而不是为了显得聪明而硬切路线
6. 允许必要的多次切换，但用户感知到的应是“被持续带着往前走”，而不是“策略在来回跳”
7. 单条回复要有明确中心，不应把人本承接、CBT 拆解、精神分析解释粗暴堆在同一段话里

## 9. 成功指标建议
### 9.1 核心漏斗
1. 进入 MoodPal 首页 -> 点击全能主理人 -> 完成隐私确认 -> 完成首轮对话 -> 完成摘要确认

### 9.2 关键指标

## 10. 对后续技术设计的直接要求
这份 PRD 至少会直接约束后续技术方案回答以下问题：
1. 如何实现“前台一个角色，后台可多流派动态路由”
2. 如何保证首轮必为人本托底
3. 如何把“人本主义层 + CBT 主轨 + 精神分析主轨 + 安全抢占层”实现成稳定的路由结构
4. 如何定义每个用户输入回合的“当轮主工作方式”，并避免同条回复里混用两种主干干预
5. 如何定义多次切入 CBT / 精神分析 / 人本修复的前置条件、切换理由与防抖约束
6. 如何实现阻抗后的降级修复
7. 如何在摘要中表达整合式推进和阶段性切换，而不暴露内部节点
8. 如何在一个会话内记录路由轨迹、切换原因与结果，又不破坏隐私承诺

## 11. 当前待确认问题
以下问题如果不先拍板，后续技术设计会出现分叉：
1. 会话结束摘要里，是否需要较明确地告诉用户“本次先后用了哪些支持方式”，还是只保留更自然的描述
2. 当系统发生多次切换时，前端是否需要在 debug 或运营视图中可见切换轨迹；用户视图当前建议不展示
3. 后续是否要支持从全能主理人会话页，一键新开某个专属角色会话；MVP 当前建议先不做
